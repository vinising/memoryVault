# Plan-First Execution Rules

## Worktree Hygiene
Before starting any implementation or instruction-authoring task, verify the git worktree is clean.
If the worktree is not clean, stop and ask the user to confirm cleanup and commit builds before continuing.

## Hard Requirement
Before executing any multi-step implementation task, always do all of the following first:
1. Generate a full implementation plan.
2. Save the plan in the repository plans/ folder.
3. Update the root agent-plan.md file with the ask summary, active plan path, status, and timestamp.

Do not begin implementation changes until the plan file exists.

## Multi-Step Threshold (Balanced)
Treat a task as multi-step if either condition is true:
1. The work has 2 or more dependent implementation steps.
2. The work changes 2 or more files.

Do not require a plan file for:
1. Pure Q&A.
2. Trivial single-step, single-file edits.
3. One-line informational responses.

## Metadata Update Cadence
When a multi-step task is active, update agent-plan.md at these points:
1. Task start.
2. After each major milestone.
3. When blockers or scope changes appear.
4. Task completion, including implementation summary.

## Metadata Format Expectations
agent-plan.md should maintain a single mutable current-task section with:
1. User ask summary.
2. Active plan file path under plans/.
3. Current status.
4. Last updated date.
5. Implementation summary when complete.

## Bugfix Timeline Index
Whenever a bug is fixed in this repository, append a concise entry to the repository timeline index in `agent-plan.md`.
Each entry should capture:
1. The bug or symptom that was fixed.
2. The file or subsystem changed.
3. The verification that confirmed the fix.
4. The date of the change.

## Plan Naming Guidance
Use date-prefixed plan names when possible:
1. YYYY-MM-DD-task-slug.md
2. Store only implementation plans in plans/.
