# GitHub Issues + PRs Workflow

## Worktree Hygiene
Before starting any implementation, verify the git worktree is clean.
If the worktree is not clean, stop and ask the user to confirm before continuing.

## Trigger Threshold
Apply this workflow when either condition is true:
1. The work has 2 or more dependent implementation steps.
2. The work changes 2 or more files.

Skip for:
- Pure Q&A or informational responses.
- Trivial single-file, single-step edits.

## Additive Override: User-Requested Feature/Fix/Test Work
For any new user request that asks to add a feature, fix a bug, or create/update/run tests (including test-only requests that run existing tests without any code changes), always create a GitHub issue first.
This override applies even when the Trigger Threshold is not met.

Use the issue as the intake artifact for each request:
1. Create a new issue in `vinising/memoryVault` before code changes.
2. Include scope, implementation plan, and verification approach.
3. Link all related commits and PRs back to that issue.

## Workflow: Issue → Branch → PR

For every qualifying user ask, follow this sequence exactly:

### Step 1 — Create a GitHub Issue (this is the plan)
The GitHub Issue replaces the local `plans/` file. The issue body IS the implementation plan.
- Use the GitHub tool to create an issue in `vinising/memoryVault`.
- Issue title: short imperative slug (e.g. "Add tag filter UI").
- Issue body must include:
  - What will be done and why (problem statement).
  - Ordered implementation steps (the plan).
  - Acceptance criteria / how it will be verified.
- Apply a label if one fits: `feature`, `bug`, `chore`, `docs`.
- Record the issue number (e.g. `#7`) — do not begin implementation until the issue exists.

### Step 2 — Work on a Branch
- Create and switch to a branch named `issue-<N>-<short-slug>` (e.g. `issue-7-tag-filter-ui`).
- Make all implementation commits on that branch.
- Commit message format: `<type>: <description> (#<N>)` (e.g. `feat: add tag filter UI (#7)`).

### Step 3 — Open a PR
- When implementation is complete and tests pass, open a PR from the branch to `main`.
- PR title: same as the issue title.
- PR body must include `Closes #<N>` so GitHub auto-closes the issue on merge.
- If the PR is a work-in-progress, prefix the title with `[WIP]`.

### Step 4 — Merge
- Merge the PR after the user confirms or after tests pass.
- The issue closes automatically.
- Delete the branch after merge.

## Mandatory Fix Review Gate (UI + Backend Side Effects)
For every bug fix, perform and document a clear regression/side-effect review before finalizing:
1. UI review: verify no visual/layout/interaction regressions in impacted views.
2. Backend review: verify no API/schema/logic/performance side effects.
3. Testing review: run targeted tests and list what was validated.
4. Document findings in the issue/PR with either:
  - "No new defects found", or
  - explicit defects/risks introduced and mitigation steps.

## agent-plan.md Usage
`agent-plan.md` is now a lightweight pointer only. Update it to reference the active issue number and branch name. Do not duplicate planning content there — GitHub is the source of truth.

## Commit Hygiene
- One logical change per commit.
- Never commit directly to `main` for qualifying tasks.
- Always run tests before opening a PR.
